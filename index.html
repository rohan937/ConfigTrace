The blind spot
Production incidents don't start in Git.

Code changes go through pull requests, code review, and CI/CD. But configuration changes bypass all of that. They happen in dashboards, CLIs, and vendor portals — with no history, no diff, and no way to trace them after the fact.


Your code has a reviewable history. Production settings do not. When something breaks, the answer is almost never in Git.

  • ×
    AWS security group opened to 0.0.0.0/0 An inbound rule widened in the console. No ticket, no review, no trail.
  • ×
    Firebase Firestore rules weakened A rule change made a collection publicly readable. Discovered days later.
  • ×
    Stripe webhook endpoint silently dropped Payment events stopped delivering. No one knew the endpoint had been deleted.
  • ×
    Cloudflare DNS record rerouted Production API traffic pointed at staging. Users started hitting errors.
  • ×
    GitHub branch protection disabled Direct pushes to main went undetected for weeks until the next incident review.
  • ×
    Supabase RLS policy removed A row-level security policy was dropped. Data exposure found during audit.
  • ×
    Vercel environment variable changed A production variable was overwritten in the dashboard. Build failures followed.

The full flow

Snapshot. Diff. Risk-assess. Repeat.

ConfigTrace treats your live infrastructure like a repository. Every sync is a commit. Every change is reviewable. Manual Sync Now is always available alongside the hourly schedule.

01 · CONNECT

Connect a provider

Add credentials for any supported provider — AWS, Firebase, Supabase, Stripe, GitHub, Cloudflare, or Vercel. Credentials are encrypted before storage and never shown again.

02 · BASELINE

Create a baseline

The first sync snapshots your current configuration — every DNS record, security rule, webhook, and policy — as your known-good baseline.

03 · SYNC

Sync on a schedule or manually

ConfigTrace syncs your connected providers on your plan's schedule. Hit Sync Now anytime to check for changes on demand, without waiting.

04 · DIFF

Detect field-level changes

Every sync is diffed against the previous state. Exactly what changed surfaces: the old value, the new value, and when.

05 · CLASSIFY

Classify change risk

Each change is scored low to critical based on blast radius. A changed TXT comment is not the same as an opened security group or a removed Firestore rule.

06 · REVIEW

Review the timeline

A clear change timeline shows what happened, when, and what it affected — so "what changed?" gets answered in seconds, across every provider.

The coverage gap

Git tracks code. ConfigTrace tracks the rest.

Your repo already has commit history, pull request reviews, and branch diffs. But production also depends on DNS records, branch protection rules, webhook URLs, repository variables, secrets metadata, and deployment settings — none of which live in a file. ConfigTrace gives those changes the same discipline Git gives code: baseline, diff, risk label, timeline.

  • → Row-level security policies and auth config (Supabase)

Data minimization

Built for metadata, not customer data.

ConfigTrace reads only configuration metadata — the rules, settings, and policies that control your infrastructure. It never reads customer records, secret values, database contents, or application data.

Risk classification

Not all changes are equal.

ConfigTrace weighs each detected change by its potential blast radius so your team can triage at a glance — not after the incident is already in progress.

Low
Domain verification TXT added
Vercel project variable added
Stripe product description updated
Hardening, ownership proof, or non-sensitive addition. No routing or security impact expected.
Medium
New subdomain CNAME created
IAM policy attached to role
Firebase auth provider enabled
Expanded access surface or altered routing. Worth a closer look before it causes issues.
High
Stripe webhook endpoint changed
Supabase RLS policy modified
GitHub branch protection weakened
Production traffic, data access, or security controls may be affected. Needs prompt review.
Critical
AWS security group opened to 0.0.0.0/0
Firestore rules allow all reads
Root DNS record deleted
Infrastructure exposed, data potentially public, or the site may be offline. Act immediately.
